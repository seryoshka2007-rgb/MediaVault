# CLAUDE.md — рабочие правила для агента

Этот файл читается автоматически в начале каждой сессии. Он кодирует ТЗ и
методологию проекта MediaVault. Соблюдай его строго.

## Что это за проект
Локальное офлайн-first приложение для личной медиатеки (Windows, позже Linux/macOS).
Полный спецификейшн — в `docs/` и `ROADMAP.md`. Приложение обязано быть полностью
работоспособным без интернета; сеть — только опциональное удобство.

## Стек
Python 3.13+, PySide6 (Qt), SQLAlchemy 2.0 + SQLite, Alembic, pydantic,
pytest / ruff / mypy, PyInstaller. Конфигурация — JSON. Логи — модуль `logging`.

## Архитектура — НАРУШАТЬ НЕЛЬЗЯ
Строгие слои, поток строго в одну сторону:

    GUI (app/) → services (core/services) → repositories (core/repositories)
               → ORM (core/models) → SQLite

Жёсткие правила:
- GUI вызывает ТОЛЬКО сервисы. Никогда — репозитории, ORM или SQL напрямую.
- Любой доступ к БД — только через `core/repositories/`. Никакого сырого SQL нигде.
- Через границу GUI↔сервис ходят только валидированные DTO (`core/schemas.py`),
  а не ORM-объекты.
- Эпизодность (сезоны/серии) определяется свойством `EntryType.is_episodic`,
  а не ветвлением `if type == SERIES`.
- Новые провайдеры источников — через протокол `Provider` + `ProviderRegistry`.
- Темы — модульные `.qss` в `themes/`.

## Безопасность (обязательно)
- Перед любым сетевым запросом провайдера — `url_validator.is_safe_to_fetch()`
  (блокирует loopback/приватные сети/метадату). НИКОГДА не фетчить без этой проверки.
- Запись файлов — только атомарно (`core/utils/files.atomic_write_*`).
- Пути от пользователя — через `ensure_within()` (защита от path traversal).
- Бэкап — только через SQLite backup API (`BackupService`), не копированием файла.
- Перед потенциально опасными операциями — обязательный бэкап.
- См. `docs/SYNC_WARNING.md`: НЕ синхронизировать живой `.db` через файловую
  синхронизацию. Синхронизировать снимок/экспорт.

## Методология (из ТЗ)
1. Разработка поэтапная; каждый этап заканчивается ПОЛНОСТЬЮ рабочей версией.
2. Каждое изменение — сначала обсуждение, потом отдельная ветка Git.
3. После этапа — рефакторинг только затронутых модулей.
4. Новые возможности интегрируются через существующие интерфейсы, без нарушения
   модульности.
5. Хорошие идеи «на потом» пиши в `ROADMAP.md` / `TODO.md`, не прерывая текущий этап.
6. PEP 8, обязательная типизация новых модулей, `from __future__ import annotations`.

## Границы (что делать НЕ надо без явной просьбы)
- Не добавлять то, что в ТЗ вынесено за v1.0 (свой сервер/API, мобильное приложение,
  сложная синхронизация, система плагинов, редактор тем, постеры).
- Не менять архитектуру ради скорости. Не плодить «заглушки» без необходимости.
- Не коммитить пользовательские данные/секреты (см. `.gitignore`).
