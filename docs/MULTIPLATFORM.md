# Мультиплатформа: iOS, Android, macOS + синхронизация

Этот документ фиксирует архитектурное решение расширить MediaVault за
границы v1.0 (см. `CLAUDE.md`, раздел "v2.0 / мультиплатформа"). Он описывает
целевую архитектуру и пофазный план; что из этого уже реализовано —
см. `ROADMAP.md`.

## Почему не просто "тот же код на всех платформах"

PySide6/Qt for Python не поддерживает iOS — это не вопрос усилий, у Qt
попросту нет iOS-бэкенда для Python-биндингов. macOS Qt поддерживает
(десктопный Qt кроссплатформенный), но собрать `.app`/`.dmg` можно только на
macOS — PyInstaller не кросс-компилирует. Физического Mac для этого не
обязательно: `.github/workflows/build-macos.yml` собирает `.app` на
GitHub-хостед macOS-раннерах (arm64 + x86_64) при пуше в `master`, готовый
файл скачивается из вкладки Actions. Android у Qt for Python есть в
экспериментальном статусе, но незрелый и тяжёлый для сборки.

Поэтому "тот же принцип работы" реализуется как **общая архитектура и
формат данных**, а не как один буквальный кодбаза на всех платформах:

| Платформа       | Стек                          | GUI-код                    |
|-----------------|--------------------------------|-----------------------------|
| Windows         | Python + PySide6 (как сейчас)  | `app/` (этот репозиторий)    |
| macOS           | Python + PySide6 (тот же код)  | `app/` — собирается на Mac   |
| iOS + Android   | Flutter/Dart (один код на оба) | `mobile/` (новый, отдельный) |

Слой `core/` (модели/схемы/сервисы/бизнес-правила) — источник истины для
Windows/macOS. Flutter-приложение реализует ту же бизнес-логику на Dart
(разделяемый код невозможен между Python и Dart напрямую), но **работает по
тому же контракту данных** — тем же полям `Entry`, тому же протоколу
синхронизации — так что поведение остаётся одинаковым, даже если код не
общий.

## Синхронизация

### Проблема идентичности записей
`Entry.id` — локальный `INTEGER PRIMARY KEY AUTOINCREMENT`, уникален только
внутри одной БД. На телефоне и на компьютере `id=1` — разные фильмы.
Для синхронизации нужен глобально стабильный ключ: `Entry.uuid` (генерируется
на клиенте при создании, `uuid4()`).

### Общий каталог vs личное состояние (Phase E)
Несколько разных людей могут пользоваться одним сервером — не только "один
человек, несколько своих устройств" (Phase A/B), но и семья/друзья с общим
списком фильмов. Поэтому сущность разделена на две:

- **Title** (каталог) — название, оригинальное название, тип, год, ссылка,
  описание. Общее для всех, кто на этом сервере. Правки — от любого;
  **удаление — только от admin** (сервер отклоняет попытку участника,
  а не просто скрывает кнопку в клиенте).
- **UserState** (личное состояние) — статус, оценка/вторая оценка,
  избранное, прогресс серий, счётчик открытий, комментарий. Привязано к
  конкретному человеку (`Person`), никогда не перезаписывается чужим при
  синхронизации.

На desktop локальная таблица `Entry` физически остаётся одной строкой
(каталог+состояние вместе — там всегда только "я"), но у неё **два
независимых таймstamp'а**: `updated_at` (любое изменение) и
`catalog_updated_at` (только правки каталожных полей). Это принципиально:
без раздельного таймстампа правка личной оценки бампала бы и "как будто
изменённое" время каталога, и push мог бы затереть чужую более свежую правку
названия каталожными данными, которые вы на самом деле не трогали.

### Идентичность людей
Один человек может иметь несколько устройств (телефон + ноутбук) с одним и
тем же личным состоянием — поэтому идентичность на уровне `Person`, не
`Device`. При регистрации участник указывает своё имя: если человек с таким
именем уже есть — новое устройство присоединяется к его существующему
состоянию; если нет — создаётся новый `Person`. Без паролей — вход по
ключу (`admin key` или `participant key`) уже ограничивает доступ,
имя — просто идентификатор внутри уже доверенного круга.

### Протокол (REST/JSON)
- `POST /devices/register` — `{key, person_name, label}`. `key == admin key`
  → всегда резолвится в единственного admin `Person` (singleton, имя из
  первой регистрации). `key == participant key` → находит/создаёт `Person`
  с ролью `participant` по имени.
- `POST /sync/push` — `{"titles": [...], "states": [...]}`. Title
  create/update — любая роль. Title с `deleted_at` — **только admin**,
  иначе сервер отклоняет (`applied=False, reason="requires admin role"`);
  клиент вообще не пытается отправить удаление каталога от лица участника.
  UserState — `person_id` всегда берётся из аутентифицированного устройства,
  что бы ни было в теле запроса — подменить чужое состояние нельзя
  структурно, не только политикой.
- `GET /sync/pull?since=<ISO-timestamp>` — `titles` (все изменения в
  каталоге от кого угодно) + `states` (только состояние **своего**
  `person_id` — чужие статусы/оценки никогда не приходят).
- **Конфликты: last-write-wins по `updated_at`, независимо для Title и
  для каждого UserState.** Не CRDT, не merge по полям — осознанно просто.
- **Хранилище сервера:** SQLite — тот же стек, что и десктоп, действительно
  "лёгкий" self-hosted сервис (можно поднять на своём NAS/VPS/Raspberry Pi).

### Статус (обновляется по ходу — актуальная сводка в ROADMAP.md)
- `sync-server/` развёрнут на сервере пользователя (Tailscale), реализует
  Title/UserState/роли, подключён к desktop-приложению.
- `core/services/sync_service.py` + кнопка "Синхронизировать" в GUI —
  диалог настройки спрашивает адрес сервера, ключ, имя человека, название
  устройства.
- macOS-сборка десктоп-приложения — через GitHub Actions CI, не требует
  физического Mac для получения `.app`.
- Мобильное (Flutter) приложение — только скелет, не реализовано.

## Пофазный план (детали — в ROADMAP.md)
- **Phase A (сделано):** дизайн-документ, `sync-server/`, `mobile/` (скелет
  Flutter, без сборки — нет SDK на этой машине).
- **Phase B (сделано):** `Entry.uuid`/`deleted_at` + миграция,
  `core/services/sync_service.py`, кнопка "Синхронизировать" в GUI.
- **Phase C:** реализация Flutter-приложения (экраны, локальная БД на
  устройстве, клиент синхронизации) — требует Flutter SDK, для iOS —
  физический Mac с Xcode всё равно нужен для сборки/подписи/публикации
  (в отличие от desktop-macOS-сборки, GitHub Actions не решает вопрос
  App Store / TestFlight подписи).
- **Phase D (сделано):** сборка macOS-версии desktop-приложения через
  `.github/workflows/build-macos.yml`. Сборка не подписана (не Apple
  Developer сертификатом) — Gatekeeper при первом запуске покажет
  предупреждение "unidentified developer", открывать через правый клик →
  "Открыть" или `xattr -cr MediaVault.app` в терминале.
- **Phase E (сделано):** разделение Title/UserState, роли admin/participant,
  `Entry.catalog_updated_at` — общий каталог фильмов с личным статусом на
  сервере с несколькими людьми.

## Ограничения, о которых нужно помнить
- iOS-сборку и тестирование Flutter-кода я не могу выполнить с этой машины
  (нет Flutter SDK, нет Xcode/macOS). Код в `mobile/` — это стартовая точка
  для сессии/среды, где эти инструменты есть, не проверенный запуском код.
- `sync-server` — сервис, слушающий сеть. Прежде чем разворачивать его
  где-либо реально доступном из интернета, нужно продумать: HTTPS
  (например, за reverse-proxy с TLS), ротацию/отзыв токенов, ограничение
  частоты запросов. Сейчас это не входит в Phase A — там только локально
  проверяемый скелет.
